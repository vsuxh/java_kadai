内容

C言語の「構造体」とJava等の「クラス」で大きく異なる機能の一つが, 「情報隠蔽」である. 情報隠蔽はオブジェクト指向プログラミングの重要な要素である.

※用語として他に「データ隠蔽」や「カプセル化」という呼び方もある.

今回はこの情報隠蔽について, これまでに作成したプログラムを拡張して体験的に学習する. 今回の内容は後期にGUI(Graphical User Interface; グラフィカル・ユーザー・インターフェース)のプログラミングを行う際に必須の基礎知識となるため, しっかり理解しておいて欲しい.

フィールドのprivate/public化とアクセサーメソッド

これまでに皆さんが作成してきたJavaプログラムでは, フィールドの値を読み取ったり書き換えたりするのは, C言語の構造体と同様に自由であり, クラスの外部からもドット演算子を用いて自由にアクセス(読み・書き)することができた. しかし, この様な自由はまた, 責任の所在, すなわちそのフィールドに関わるコードをソース中のあちこちに分散させ, バグの追求を困難にしてしまう. 例えば,

class Koza // 銀行の預金口座(機能を提供する側)
{
    double riritsu;
    int gankin;

    Koza() {
        riritsu = 0.002; // 利率は銀行側で決めたいのに, 
    }
}

class KozaApp // 銀行の顧客(機能を使わせてもらう側)
{
    public static void main(String[] args) {
        Koza my_koza = new Koza(); // 口座を開設した後
        my_koza.riritsu = 77.7;    // 勝手に利率を変更し放題!!
        double kotae = my_koza.risoku(6); // 勝手に変更したその利率で利息を計算
    }
}

そこでオブジェクト指向プログラミングでは, privateの壁でフィールドを外部から「隠蔽」して守り, フィールドに直接アクセス(読み・書き)するコードをそのクラスの内部に限定する. こうすることで, 外部に見せたくないフィールドを隠したり, フィールドの値を変更する(つまり責任を伴う)コードをこのクラスの中に限定できる. この, フィールドを隠す・隠さないを指示する命令(修飾子)が, 「private」と「public」である. 

class Koza
{
    private double riritsu; // private化
    private int gankin;     // private化. これにより,

    Koza() {
        riritsu = 0.002; // クラス内部では自由にアクセスできるが
    }
}

class KozaApp
{
    public static void main(String[] args) {
        Koza my_koza = new Koza();
        my_koza.riritsu = 77.7; // クラスの外部からはアクセスできなくなる
    }
}

ただし, gankinの様に, 利用者にもある程度の変更の自由を与えたい場合は, 完全に隠蔽しては使い物にならない. そこで, フィールドの値を読み取ったり書き換えたりできるメソッドを用意し, 外部からはそのメソッドを呼び出して使わせる. この, privateによる壁の出入り口となるメソッドを, 「ゲッター」(getter)「セッター」(setter)と呼び, 両者を合わせて「アクセサー」(accessor)と呼ぶ.

※アクセサ「リー」ではないので注意. 

class Koza
{
    private double riritsu; // privateフィールド
    private int gankin;     // privateフィールド

    Koza() {
        riritsu = 0.002;    // 利率は銀行側で設定
    }

    int getGankin() {  // 元金のゲッター
        return this.gankin; // 値を返すだけ
    }
    void setGankin(int gankin) { // 元金のセッター ※戻り値の型 void に注意
        if (gankin >= 1 && gankin <= 10000000) { // 1円以上1千万円までに制限(ある程度の自由)
            this.gankin = gankin;                // 条件を満たした場合のみフィールドに代入
        }
    }

    double getRiritsu() { // 預金口座の性質上, 利率はゲッターのみの提供とする
        return this.riritsu;
    }
    // 利率のセッターを作らない == セット機能を提供しない == 外部からセットさせなくする
}

class KozaApp
{
    public static void main(String[] args) {
        Koza my_koza = new Koza();
        System.out.println("利率は" + my_koza.getRiritsu()); // 顧客は利率を知ることはできる
        my_koza.setGankin(68000); // 元金に直接代入はできないので, セッターを呼び出して使う
    }
}

この様に情報隠蔽を行うことで, クラスを作る側(機能を提供する側)と使う側(機能を使わせてもらう側)を明確に分離でき, 大規模で複雑なソフトウェアの見通し良い作成や管理が可能となる. この情報隠蔽によるアクセス制限の考え方は, 現在では各種ソフトウェアを作成する際には必須の知識となっている.
