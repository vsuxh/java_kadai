MVCモデル

MVCモデルは, オブジェクト指向プログラミング(OOP; Object-Oriented Programming)においてクラス分割を行う際に基準となる考え方の一つである. MVCモデルではプログラム全体を以下の三つの役割のクラスに分割して作成する. ※それぞれ更に細かく分割しても良い.

  • モデル(Model) ... データの保存や計算を担当する.
  • ビュー(View) ... データの表示を担当する.
  • コントローラー(Controller) ... モデルとビューの制御を担当する.

図: MVCモデル

このMVCモデルを, Java SwingのGUIアプリの構造に当てはめると, 大枠で以下の通りのクラス分割が考えられる. (イベント処理など, どうしてもMVCの通りには分割できない場合もある)

  • モデル(Model) ... GUIではない, 前期プログラミング言語IIIAで作成した様な基本的なクラス.
  • ビュー(View) ... 描画パネル. グラフィックスの描画を担当する.
  • コントローラー(Controller) ... フレーム. GUI部品を生成したり配置したりタイマー制御を担当する.

モデルクラスの役割が分かりにくい場合は, ゲーム等のセーブデータを思い浮かべると良い. そこに現在の場面を再現するのに必要なすべての情報が記録されている, と考えてみよう. セーブする際はモデルをファイルに保存し, ロードする際はファイルからモデルを読み込んでそれをビューが表示している. ※今回の課題で実際にセーブ/ロード機能を作成する必要はない.

MVCに分割してプログラムを作成することで, 各部分の独立性を高められ, 大規模なプログラムの見通しが良くなると共に, 更新や再利用等の管理も容易になる.

  • 自作のクラスで定数を定義する場合は, 「public static final 型 定数名 = 値;」とする. なお, 定数名はすべて大文字+アンダースコアで付ける(Java標準命名規則).
    • public class MyFrame extends JFrame {
    public static final int NUM_BALLS = 5; // ボールの個数
}
  • 標準ライブラリで用意されている可変長の配列を使いたい場合はArrayListクラスを調べてみると良い.
  • sin関数などの数学関数は, Math.sin()の様に先頭に「Math.」を付加する事で利用できる. なお, 引数の単位はラジアンである. 円周率πには定数Math.PIを利用できる.

課題

今回はレポート提出システムによるアップロード方式ではなく, 学内ファイルサーバーの学内ウェブ公開用ディレクトリに jar ファイルや html ファイルをコピーすることで提出とする.

開発中は従来通り, 今回のテーマの課題遂行用ディレクトリで作業し, そこでjarファイル化して動作確認を行った後, 公開用ディレクトリへコピーすると良い. ファイル名やディレクトリ名に不備があると自動回収が不可能となり, 未提出として扱われるため, ディレクトリ名やファイル名が指定された通りになっているか, 良く確認すること.

【課題】これまでの多数の課題遂行により得られた知識をフル活用し, Java SwingでGUIアプリを作成せよ. 以下の条件を満たす様に作成すること.

  • グラフィックス描画とタイマーを用いてアニメーションさせる.
  • [重要] クラスはMVCモデルに従って三つに分割する. さらに分けて三つ以上でも良い.
  • [重要] 各クラスにコンストラクタやゲッター, セッターを用意し, クラス間でデータの受け渡しを行う. 抜け道となるpublicフィールドや, staticフィールドは原則として禁止する. (定数定義で使うのは可)
  • 少なくとも, 前回テーマまでで作成したアニメーションを元にして, データ(chara_i, chara_x, chara_y等)を抜き出して別クラスにし, ゲッター/セッターを付けて, モデルクラスとすること.
  • 上級者向け: 必ずしも画像ファイルを使う必要はなく, 計算で求めた(生成した)グラフィックスを時間的に変化させる内容でも良い. また, キーボードイベントやマウスイベントを活用しても良い. キーボードイベントやマウスイベントのイベント処理は, Swingの仕組み上, やむを得ず描画パネルに付ける.
  • 上級者向け: フレーム側でモデルクラスに update をかけて, 描画パネルを repaint しても良い.
  • 公開用ディレクトリは各自の ~/public_html/java/app22 とする.
  • jarファイルのファイル名は「ユーザーID.jar」とする. ※今年度の皆さんのユーザーIDはおおむね「p22xxxx」
  • HTMLファイルの名前は ユーザーID.html とする.
    • ページの最初に出席番号・名前・テーマ番号を記入する.
    • 自動回収の都合上, 複数のHTMLファイルには分割せず, ユーザーID.html ひとつで作成して欲しい.
    • CSSやJavascriptを使用して装飾しても良いが, 別ファイルになっていると自動回収は保証できないので, 埋め込みを推奨する. (別ファイルかつ学外のファイルは論外. 学外のファイルを指定しないこと.)
  • HTMLファイルからソースファイルを閲覧できるようにする. ソースファイルを ユーザーID.html と同じ場所に置き, ユーザーID.html からリンクする. ※どうしても非公開にしたい場合は教員に相談・確認の上, 直接提出すること.
  • ディレクトリ構成は以下の通り.
    • ~/public_html/java/app22/ の直下に
         ユーザーID.html  (ここから各ファイルへリンクする)
         ユーザーID.jar   (この中にマニフェストファイルとクラスファイルと画像ファイルを入れる)
         MyFrame.java     (閲覧用) ソースファイル名はこの通りでなくても良い
         MyPanel.java     (閲覧用) ソースファイル名はこの通りでなくても良い
         MyModel.java     (閲覧用) ソースファイル名はこの通りでなくても良い
  • ユーザーID.html に説明を記入する. スクリーンショットが入るとなお良い.
    • ユーザー向けの説明 ... アニメーションの内容や, (もし操作があれば)操作方法の説明
    • 開発者向けの説明 ... アニメーションの仕組みの説明. どの様に座標等の変数を変化させているか.
    • クラス分割の説明 ... どの様にクラス分割を行ったか. 何のデータを, いつ, どの様に, どのクラスからどのクラスへ, 受け渡ししているか.
  • [注意] 成績保管のため, 今回作成したウェブページは消去せずに残しておくこと.
  • 主な評価項目
    1. 所定のディレクトリに配置されており, 他の人がダウンロードして動作させることが可能であるかどうか
    2. 三つの説明文(ユーザー向け, 開発者向け, クラス分割)
    3. jarファイルやソースファイルへのリンク
